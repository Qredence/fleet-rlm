# Ephemeral Memory Protocol

## Overview

The Ephemeral Memory Protocol is a markdown-based strategy for managing "working memory" during an AI interaction. It prevents the context window from being overwhelmed by raw history while ensuring that critical transient data—like current task status, temporary file paths, or specific debugging findings—remains immediately accessible.

**Core principle:** Keep the context window lean by distilling transient facts from the interaction history into a structured, self-updating block that is cleared once the current sub-task is complete.

## Capabilities

- **Session State Tracking** — Maintain variables and flags across 5-20 turns.
- **Context Distillation** — Extract essential facts from verbose logs or chat history.
- **Markdown-Based Storage** — Use standard headers and lists for easy LLM parsing.
- **Self-Cleaning Lifecycle** — Automatically prune or overwrite data as tasks evolve.

## Dependencies

- `_core/state_management` — For the logic of updating and persisting keys.
- `mcp_capabilities/context_management` — For injecting the block into the system prompt or top-level message.

## When to Use

**Use when:**
- You are debugging a complex issue involving multiple temporary files or error traces.
- You need to track the progress of a multi-step plan where steps depend on specific outputs of previous steps.
- You have transient user preferences (e.g., "For this session, use tabs instead of spaces").
- You want to "compress" a long conversation history into just the current "state of play."

**When NOT to use:**
- For permanent project rules (use `project_context` instead).
- For long-term user memories (use a dedicated `user_profile` or database).
- For storing secrets or credentials.

## Quick Reference

| Problem | Solution | Keywords |
| ------- | -------- | -------- |
| Context window overflow | Distill history into Ephemeral Memory | compression, token-saving |
| Repeating previous errors | Log "Attempted Solutions" in Ephemeral Memory | failure-tracking, state |
| Losing track of file paths | Store "Current Working Files" block | navigation, path-tracking |

## Core Patterns

### The Ephemeral Memory Block

**The problem:** Vital information is lost as it slides out of the LLM's effective attention span (middle of the context window).

**❌ Common mistake (Implicit reliance on history):**
Relying on the LLM to "remember" a file path mentioned 10 messages ago.
```text
(User: 10 turns ago) "The log is at /tmp/error_99.log"
...
(Agent) "I can't find the log path, could you provide it again?"
```

**✅ Production pattern (Explicit memory block):**
Maintain a block at the top of the interaction or within the system prompt.

```markdown
### Ephemeral Memory
- **Current Task:** Debugging connection timeout in `AuthService`.
- **Target File:** `/src/services/auth.ts`
- **Latest Trace:** `ETIMEDOUT at line 45`
- **Attempted:** Restarted docker, checked env vars.
- **Session Preference:** Use verbose logging in responses.
```

**Key insight:** By explicitly rewriting this block at the end of a turn, the agent "re-primes" its own context for the next turn, ensuring the most relevant data is always at the "top of mind."

## Identification Logic: What belongs where?

| Data Type | Destination | Life Span |
| :--- | :--- | :--- |
| **Transient** (Temp paths, loop counts, current error) | `Ephemeral Memory` | 2-10 Turns |
| **Persistent** (Coding standards, project architecture) | `Project Context` | Permanent |
| **Historical** (Raw user messages, tool outputs) | `Interaction History` | Until Window Limit |

## Lifecycle Operations

1.  **Initialization**: Create the `### Ephemeral Memory` block upon the first multi-step request.
2.  **Update/Overwrite**: During the "Response Generation" phase, update specific lines. If a "Target File" changes, overwrite the old one—do not create a list of previous files.
3.  **Garbage Collection**: 
    *   **Trigger:** On "Task Complete" signal.
    *   **Action:** Delete the specific item or clear the entire block.
    *   **Trigger:** If the block exceeds ~20 lines.
    *   **Action:** Distill 20 lines into 5 summary bullets.

## Usage Examples

### Debugging Session State

Tracking the state of a complex investigation into a failing test suite.

```markdown
### Ephemeral Memory
- **Status:** Investigating `Test_User_Auth` failure.
- **Active Breakpoint:** `user_controller.py:112`
- **Working Hypothesis:** JWT secret is not being loaded from `.env.test`.
- **Verified:** `.env.test` exists; `os.getenv` returns `None`.
```

### Code Refactoring State

Managing a multi-file migration where global consistency is required.

```markdown
### Ephemeral Memory
- **Refactor Goal:** Migrate class-based components to Functional Components with Hooks.
- **Completed Files:** `Header.tsx`, `Footer.tsx`.
- **Current Focus:** `Sidebar.tsx`.
- **Remaining:** `MainContainer.tsx`, `UserProfile.tsx`.
- **Pattern Note:** Ensure `useContext` is used for the 'Theme' provider instead of HOCs.
```

## Common Mistakes

| Mistake | Why It's Wrong | Fix |
| ------- | -------------- | --- |
| Accumulating History | If you keep adding bullets without deleting, it becomes a second history log. | **Overwrite** keys; keep the block under 10-15 lines. |
| Storing Secrets | Ephemeral memory might be logged or cached in plain text. | Never put API keys or passwords in the memory block. |
| Mixing with Context | Putting permanent project paths in ephemeral memory. | Move static info to a `.project-context.md` file. |

## Strong Guidance

- **NO DUPLICATION** — If a value is updated, replace the old line; do not append.
- **ALWAYS READ FIRST** — Every turn must begin by parsing the Ephemeral Memory to set internal state.
- **NEVER EXCEED 500 TOKENS** — Ephemeral memory is for agility; if it's large, it's no longer ephemeral.
- **MANDATORY CLEANUP** — Clear variables as soon as the associated sub-task is marked "Done."

## Red Flags

- The `### Ephemeral Memory` block contains the same information as the `Interaction History` without any simplification.
- The agent asks for information that is currently present in the Ephemeral Memory block.
- The block has not been updated in 5+ turns despite a change in the task.
