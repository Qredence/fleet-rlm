## Common Mistakes

| Mistake | Why It's Wrong | Fix |
| ------- | -------------- | --- |
| Overusing `# type: ignore` | Masks bugs rather than defining types. | Use `Any` explicitly or fix the underlying type logic. |
| Global `strict = true` on legacy | Causes immediate pipeline failure on unrelated code. | Use `include`/`exclude` or directory-specific overrides. |
| Not using `uv run` | Can lead to "Module Not Found" errors if `ty` uses a different Python env. | Always prefix execution with `uv run`. |
| Ignoring `pyproject.toml` | Default settings might be too loose for production safety. | Explicitly define `python_version` and `target` in config. |

## Strong Guidance

- **NO NEW UNTYPED CODE** — Every new file added to the project MUST be included in a `strict` override.
- **ALWAYS USE UV** — Ensure `ty` version consistency across the team by managing it via `uv` lockfiles.
- **NEVER DISABLE IMPORT-CHECKING GLOBALLY** — Disabling import checks makes the type checker "blind" to external API changes.
- **FIX > IGNORE** — A `# type: ignore` is a technical debt check that should require a comment explaining *why* it is necessary.

## Red Flags

- The `pyproject.toml` has no `[tool.ty]` section (using defaults is rarely correct for teams).
- Developers are complaining that `ty` is "getting in the way" (indicates rules are too strict for the current codebase maturity).
- The CI step for `ty` takes longer than 30 seconds for a medium-sized project (check for recursive directory issues or symlinks).
