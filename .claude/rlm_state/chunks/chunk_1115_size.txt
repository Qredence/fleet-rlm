## Dynamic Skill Mounting Strategy

### 1. **Planner Agent Workflow**
```
Input: User Task
  ↓
Parse Task Intent
  ↓
Identify Required Skill Branches
  ↓
Calculate Skill Dependencies
  ↓
Optimize Skill Set (minimize overhead)
  ↓
Mount Skills to Worker Agent(s)
  ↓
Monitor & Adapt
```

### 2. **Skill Selection Criteria**
- **Primary Skills**: Direct task requirements
- **Supporting Skills**: Dependencies and enablers
- **Context Skills**: Project-specific knowledge
- **Adaptive Skills**: Based on interaction history

### 3. **Mounting Levels**
- **Level 1 - Core**: Always loaded (reasoning, basic I/O)
- **Level 2 - Task-Specific**: Mounted for current task
- **Level 3 - Contextual**: Available on-demand
- **Level 4 - Dormant**: Cached but not loaded

### 4. **State Management**
```json
{
  "session_id": "unique_session",
  "mounted_skills": [
    "COGNITIVE_SKILLS.Analysis.Code_Analysis",
    "TECHNICAL_SKILLS.Programming.Python",
    "TOOL_PROFICIENCY.Development_Tools.Version_Control"
  ],
  "context_stack": [
    {
      "task": "debug_authentication",
      "active_memory_blocks": ["Project_Context", "Interaction_History"],
      "skill_usage_metrics": {}
    }
  ],
  "evolution_log": []
}
```

 Restructure Skills Fleet Codebase

 Goal

 Comprehensive restructuring of Skills Fleet to align with task-driven organization principles established in our dspy_modules_library, with FastAPI as the main application that uses DSPy components
 to create skills, manage adaptive taxonomy, and provide user-specific broad skills.

 Requirements

 - Primary Goal: All of the above (apply library organization, improve discoverability, better separation of concerns)
 - Scope: Comprehensive (entire codebase: api/, cli/, core/, and all supporting areas)
 - Alignment: Yes, but adapt them (use similar categories adapted for Skills Fleet's broader scope)

 Critical Architecture Requirement

 FastAPI is the main application - not just one of several interfaces. The FastAPI app should:
 - Use DSPy components (signatures, modules, programs, optimizer, evaluations, tools)
 - Replicate the skill-fleet chat CLI interactive functionality via API endpoints
 - Manage and maintain an adaptive taxonomy
 - Provide broad skills that are specific to user query/needs
 - CLI becomes an alternative interface to the same core functionality

 Agent Skills Specification Integration

 Based on analysis of the Agent Skills knowledge base (via RLM), the restructuring must align with the official Agent Skills specification:

 Agent Skills Structure (from agentskills.io spec)

 Required Format:
 skill-name/
 ├── SKILL.md                    # Main skill file (under 500 lines)
 ├── scripts/                    # Optional executable scripts
 │   └── *.py, *.sh, *.js
 ├── references/                 # Optional progressive disclosure
 │   ├── FORMS.md               # Form templates
 │   ├── REFERENCE.md           # Detailed reference material
 │   └── examples.md            # Usage examples
 └── assets/                    # Optional supporting files

 SKILL.md Frontmatter (YAML):
 ---
 name: skill-name                    # Required: max 64 chars, lowercase/hyphens
 description: Processes X for Y...   # Required: max 1024 chars, third-person
 license: mit                        # Optional
 compatibility: [claude-code, api]   # Optional
 metadata:                           # Optional
   author: ...
   version: ...
 allowed-tools: [...]                # Optional: tool restrictions
 ---

 Key Constraints:
 - Name: lowercase, numbers, hyphens only (no consecutive hyphens)
 - Description: Must describe WHAT skill does AND WHEN to use it
 - Body: Under 500 lines for optimal performance
 - Paths: Use forward slashes (Unix-style) on all platforms

 Taxonomy Gap Identified

 Critical Finding: The official Agent Skills specification has no built-in taxonomy system. Skills are standalone filesystem directories with no categorization or relationship management. This is
 exactly what Skills Fleet's adaptive taxonomy system will provide as a value-add layer.

 Skills Fleet Value-Add Layers

 Beyond the base Agent Skills spec, Skills Fleet adds:

 1. Adaptive Taxonomy System
   - User-specific skill categorization
   - Dynamic taxonomy based on usage patterns
   - Hierarchical skill organization
   - Taxonomy queries via FastAPI endpoints
 2. DSPy-Powered Skill Generation
   - Automated skill creation from natural language
   - HITL-guided skill refinement
   - Quality validation pipeline
   - Iterative improvement loops
 3. Skill Management API
   - CRUD operations for skills
   - Version management (epoch timestamps, 'latest')
   - Dependency tracking between skills
   - Skill composition and bundling
 4. Multi-Interface Support
   - FastAPI REST endpoints
   - CLI chat interface (same workflows)
   - Consistent skill behavior across interfaces

 Current Structure Issues

 1. Mixed concerns in core/dspy/: Signatures, modules, metrics, monitoring, training all mixed together
 2. Unclear boundaries: Conversation logic appears in both modules/ and services/
 3. Phase-based naming: phase1_understanding, phase2_generation, phase3_validation - describes WHEN not WHAT
 4. Scattered HITL logic: Found in core/hitl/, cli/hitl/, and throughout modules/
 5. No clear entry points: Difficult to find where workflows begin/end

 Key Principles Applied

 1. FastAPI-Centric Architecture

 - app/: FastAPI is the main application, not just one of several interfaces
 - API endpoints: Directly expose workflow orchestrators via REST
 - CLI as alternative: CLI uses the same workflows through app/services/ layer
 - Single source of truth: Both API and CLI call the same workflow orchestrators

 2. Task-Driven Organization

 - workflows/: Orchestrators by WHAT THEY DO (conversational_interface, task_analysis_planning, etc.)
 - dspy/signatures/: Organized by task, not source file
 - dspy/modules/: Compose signatures into reusable modules

 3. Separation of Concerns

 - app/: FastAPI application (main entry point)
 - app/api/: Route definitions and schemas
 - app/services/: FastAPI service layer (bridges API to workflows)
 - cli/: Alternative interface using same workflows
 - workflows/: Orchestration layer (glues DSPy components together)
 - dspy/: DSPy-specific logic (signatures, modules, metrics, programs, optimizer, evaluations, tools)
 - domain/: Business logic independent of DSPy
 - services/: Cross-cutting technical services
 - infrastructure/: Technical concerns (config, database, messaging)

 4. Clear Boundaries

 - DSPy logic stays in dspy/
 - Business logic in domain/
 - Orchestration in workflows/
 - FastAPI app logic in app/
 - CLI in cli/ (alternative interface)

 5. Improved Discoverability

 - Descriptive, action-oriented names
 - Group by function, not by phase or type
 - Clear hierarchy: app (FastAPI) → app/services → workflows → dspy → domain

 6. Adaptive Taxonomy Support

 - domain/taxonomy/manager.py: Core taxonomy management
 - domain/taxonomy/adaptive.py: User-specific taxonomy adaptation
 - app/api/v1/taxonomy/: Taxonomy management endpoints
 - app/services/taxonomy_service.py: Service layer for taxonomy operations

 Skill Generation Workflow (Best Practices)

 Based on the Agent Skills specification analysis, the DSPy-powered skill generation will follow these best practices:

 Generation Principles

 1. Conciseness First
   - Challenge each piece of information
   - Assume Claude is already smart
   - Only add context it doesn't have
   - Target: Under 500 lines for SKILL.md body
 2. Progressive Disclosure
   - SKILL.md as high-level overview
   - Move detailed material to references/
   - FORMS.md for templates
   - REFERENCE.md for detailed docs
   - examples.md for usage examples
 3. Match Specificity to Task Fragility
   - High freedom: Text-based instructions (multiple valid approaches)
   - Low freedom: Fragile operations requiring exact sequences
 4. "Solve, Don't Punt"
   - Scripts should solve problems explicitly
   - Include helpful error messages
   - Don't just punt back to Claude

 Skill Content Structure

 Required SKILL.md Sections:
 1. Core Principle (one-sentence foundational principle)
 2. When to Use (triggering conditions and symptoms)
 3. Quick Start (copy-paste ready examples)
 4. Core Patterns (❌ anti-patterns + ✅ production patterns)
 5. Real-World Impact (measurable outcomes)
 6. Validation (CLI commands to verify understanding)

 Quality Indicators:
 - Iron Law style guidance (ALWAYS/NEVER/MUST)
 - Paired Good/Bad examples
 - Strong imperative language
 - Concrete, executable code

 Skill Generation Pipeline (DSPy-Driven)

 User Request (Natural Language)
     ↓
 [Task Analysis - DSPy Signatures]
     ├─ GatherRequirements
     ├─ AnalyzeIntent
     ├─ FindTaxonomyPath
     ├─ AnalyzeDependencies
     └─ SynthesizePlan
     ↓ (HITL: Clarifying Questions)
 [Content Generation - DSPy Signatures]
     ├─ GenerateSkillContent
     ├─ GenerateSkillSection (for progressive disclosure)
     └─ GenerateCapabilityImplementation
     ↓ (HITL: Preview & Feedback)
 [Quality Assurance - DSPy Signatures]
     ├─ ValidateSkill
     ├─ AnalyzeValidationIssues
     ├─ AssessSkillQuality
     └─ GenerateAutoFix
     ↓ (HITL: Fix Approval)
 [Skill Complete - Ready for Deployment]

 Skill Output Validation

 The skill generation process will validate against:
 - Agent Skills spec compliance: Frontmatter, naming, description
 - Content quality: Core principle, contrast patterns, strong guidance
 - Structure: Proper heading hierarchy, working internal links
 - Token efficiency: Under 500 lines, progressive disclosure
 - Best practices: Iron Law guidance, concrete examples

 Skill Storage & Retrieval

 Storage Format:
 - Agent Skills compliant filesystem structure
 - YAML frontmatter with metadata
 - Under 500 lines SKILL.md body
 - Optional scripts/, references/, assets/

 Retrieval via Taxonomy:
 - User-specific taxonomy adaptation
 - Hierarchical categorization
 - Dependency-aware loading
 - Version management (epoch timestamps)
