## Anatomy of a Minimal Skill

Every skill MUST contain three primary components to be considered "production-ready" within the system:

1.  **The Manifest (`metadata.json`):** The identity card of the skill. It defines the `skill_id`, versioning, and dependencies.
2.  **The Documentation (`SKILL.md`):** The manual for the Agent. It uses "Common Skill Optimization" (CSO) to tell the agent *when* and *how* to use the skill.
3.  **The Logic Directory:** Contains the executable code (e.g., Python scripts or JavaScript modules).

---

## Core Patterns

### Pattern 1: Naming Conventions
❌ **NEVER** use CamelCase or spaces in skill names (e.g., `HelloWorld` or `hello world`).
✅ **ALWAYS** use kebab-case for directory names and `skill_id` (e.g., `hello-world`).

### Pattern 2: Dependency Declaration
❌ **NEVER** rely on "hidden" dependencies that aren't listed in `metadata.json`.
✅ **ALWAYS** explicitly list dependencies, even for core communication modules, to ensure proper load ordering.

### Pattern 3: Response Structure
❌ **NEVER** return a raw, unformatted string (e.g., `return "Hello"`).
✅ **ALWAYS** return a structured dictionary or JSON object to maintain compatibility with `_core/communication`.

### Pattern 4: Directory Depth
❌ **NEVER** nest skill logic deep within unrelated folders (e.g., `skills/misc/test/hello/`).
✅ **ALWAYS** follow the `skills/{category}/{skill-name}/` flat hierarchy for discovery efficiency.

### Pattern 5: Metadata Descriptions
❌ **NEVER** provide vague descriptions like "A test skill."
✅ **ALWAYS** provide triggering conditions in the description (e.g., "Use when verifying system integrity...").

---

## Real-World Impact

Implementing and maintaining a `hello-world` baseline provides measurable benefits:
- **Reduction in Bootstrap Time:** New developers can clone the atomic skill and start coding in < 60 seconds.
- **System Readiness Signal:** Automating a "Hello World" check after CI/CD deployments ensures zero-configuration failure detection.
- **Standardization Compliance:** Forces all new skills to adopt the correct directory and metadata patterns from day one.

---

## Validation

Verify your `hello-world` installation using the following CLI commands:

### Skill Discovery
Confirm the system sees the skill and its metadata:
```bash
agent skills list --filter _core/hello-world
```

### Logic Execution
Run the primary function to verify the runtime environment:
```bash
agent call hello-world:hello
```

### Metadata Integrity
Validate that the `metadata.json` conforms to the system schema:
```bash
agent skills validate _core/hello-world
```

# Design Systems Architecture

Design Systems are the connective tissue between design and engineering. They transform visual intent into a systematic, reusable, and version-controlled library of tokens and components.

## Core Principle: The Single Source of Truth (SSoT)
A Design System is NOT a UI kit or a Figma file. It is a **shared language** where every visual decision (color, spacing, typography) is abstracted into a **Token** and every UI element is built as a **Component** following Atomic Design principles. 

**SSoT Rule:** If a value (like `#3B82F6`) appears in code more than once without a token reference, the system has failed.

## When to Use This Skill
- **UI Fragmentation:** You have 14 different shades of "primary blue" across the application.
- **Redundant Engineering:** Developers are rebuilding the same "Button" or "Card" component in every new feature.
- **Scaling Pains:** Onboarding new designers/developers takes weeks because there are no established UI patterns.
- **Theming Requirements:** You need to support Dark Mode or multi-tenant branding efficiently.

## Quick Start: The Systemization Workflow
1.  **Inventory:** Audit existing UI to find "snowflakes" (one-off styles).
2.  **Tokenize:** Define a hierarchical naming convention for primitive values.
3.  **Atomize:** Build foundational components (Atoms) before complex layouts.
4.  **Document:** Write usage guidelines that explain *why* and *how* to use elements.
5.  **Sync:** Implement a handoff process that links Figma components to code modules.

---

## Core Patterns

### 1. The Token Hierarchy
Tokens must be structured to allow for global changes without breaking specific component logic.

❌ **Anti-Pattern: Hardcoded or Primitive-only Tokens**
```json
// Hardcoded
"color": "#3B82F6" 

// Primitive-only (Better, but lacks context)
"color-blue-500": "#3B82F6" 
```

✅ **Production Pattern: Semantic Alias Strategy**
```json
{
  "primitive": { "blue-500": { "value": "#3B82F6" } },
  "alias": {
    "action-primary-default": { "value": "{primitive.blue-500}" },
    "border-focus": { "value": "{primitive.blue-500}" }
  }
}
```
*Why? If you change the brand color, you update the primitive. If you want to change only the focus border color, you update the alias without affecting buttons.*

### 2. The Component State Matrix
Components MUST be defined by their states, not just their default appearance.

❌ **Anti-Pattern: Happy-path Components**
Designing a button that only accounts for the "Default" state, leaving developers to guess hover/active/focus behaviors.

✅ **Production Pattern: Full State Definition**
Every interactive component must define:
- **Default**: The resting state.
- **Hover/Pointer**: Feedback for interaction.
- **Active/Pressed**: Confirmation of intent.
- **Focus**: Accessibility requirement for keyboard navigation.
- **Disabled**: Non-interactive state (must meet contrast/logic standards).
- **Loading**: Asynchronous state to prevent double-submissions.

### 3. Atomic Structure (The hierarchy of composition)
Organize your library to maximize reusability through composition.

| Level | Definition | Examples |
| :--- | :--- | :--- |
| **Atoms** | The smallest functional units. Cannot be broken down further. | Buttons, Inputs, Icons, Labels. |
| **Molecules** | Groups of atoms functioning together as a unit. | Search bar (Input + Icon + Button). |
| **Organisms** | Complex UI sections composed of molecules/atoms. | Header, Sidebar, Data Table. |
| **Templates** | Page-level layouts showing component placement. | Dashboard Layout, Auth Page. |

---

## The UI Audit & Inventory (Phase 1)
Before building, you must catalog the chaos. 
1.  **Screenshot everything:** Collect every unique button, input, and header from the current app.
2.  **The "Snowflake" Test:** Group similar items. If you have 5 variations of a "Close" icon, identify the one that should be the standard.
3.  **Categorization:** Use an **Audit Matrix** to prioritize migration:
    *   **High Priority:** Elements used on every page (Buttons, Typography, Grid).
    *   **Low Priority:** Elements used in niche edge cases (Complex charts, one-off settings modals).

## Documentation & Governance
A system is only as good as its documentation. Every component page MUST include:
- **Anatomy:** Visual breakdown of the component's parts.
- **Props/API:** Technical specification for developers.
- **Content Guidelines:** Instructions for copy (e.g., "Button text must be < 3 words").
- **Accessibility (A11y):** Screen reader behavior and keyboard shortcuts.

---

## Real-World Impact (KPIs)
A successful Design System should move these metrics:
- **Component Reusability:** Percentage of UI built using library components vs. custom CSS (Target: >90%).
- **Development Velocity:** Time-to-market for a new feature layout (Typically reduced by 30-50%).
- **Visual Regression:** Number of UI bugs reported per sprint (Target: Near zero for core components).
- **Design Handoff Time:** Time spent by developers clarifying design specs (Reduction in Slack/Meeting overhead).

## Validation: Checking System Integrity
Use these commands to verify if the system is being bypassed in the codebase.

**Find hardcoded HEX values (The "Snowflake" Hunt):**
```bash
grep -rE "#([A-Fa-f0-9]{6}|[A-Fa-f0-9]{3})" ./src --exclude-dir=node_modules
```

**Count usage of a specific component vs. raw HTML tags:**
```bash
# How many raw buttons vs. System Buttons?
grep -c "<button" ./src
grep -c "<Button" ./src
```

**Validate Token coverage:**
Ensure that your CSS/SASS files do not contain raw pixel values for spacing.
```bash
grep -rE "[0-9]+px" ./src/styles
```

# Hello World: The Atomic Skill

The `hello-world` skill represents the minimal unit of agent capability. It serves as the baseline for system verification and the blueprint for all future skill development.

## Core Principle: The Atomic Skill
**A skill is not just code; it is a structured package of metadata, documentation, and logic.** The "Atomic Skill" principle dictates that even the simplest functionality must adhere to the standardized directory layout to be discoverable and executable by the Agentic System.

---

## When to Use

Use this skill when:
- **Validating Installation:** You have just installed the agent framework and need to confirm the environment is correctly configured.
- **Bootstrapping Development:** You are starting a new skill and need a "clean slate" template that follows current standards.
- **Testing Skill Loading:** You need to verify that the system can correctly parse `metadata.json` and index `SKILL.md`.
- **Debugging Paths:** You are troubleshooting directory permissions or skill discovery logic.

---

## Quick Start

To manually create the atomic skill, establish the following structure in your `skills/` directory:

```bash
mkdir -p skills/_core/hello-world
touch skills/_core/hello-world/metadata.json
touch skills/_core/hello-world/SKILL.md
touch skills/_core/hello-world/main.py
```

### 1. Define the Manifest (`metadata.json`)
```json
{
  "skill_id": "_core/hello-world",
  "name": "hello-world",
  "version": "1.0.0",
  "type": "technical",
  "dependencies": ["_core/communication"]
}
```

### 2. Implement the Logic (`main.py`)
```python
