# shadcn-custom-registry

## Overview

Traditional UI libraries are distributed as compiled NPM packages. This creates a "black box" where styling is difficult to override and version updates are risky. The **shadcn registry pattern** flips this model: you host a JSON manifest (the registry) that points to source files. Consuming projects use the shadcn CLI to "add" these components, downloading the source code directly into their own `/components/ui` folder.

This skill covers the architecture of building, automating, and distributing a custom organizational registry.

**Core principle:** Treat UI components as source-controlled assets rather than immutable binary dependencies, enabling high flexibility with centralized standards.

## Capabilities

- **Architect a Registry Workspace:** Structure a Vite-based repository optimized for component discovery.
- **Define Registry Schemas:** Create valid `registry.json` manifests that specify dependencies, registry hooks, and file mappings.
- **Automate Registry Generation:** Write TypeScript scripts to dynamically generate the registry based on file-system metadata.
- **Manage Cross-Component Dependencies:** Link components within the registry so that adding a high-level "DataGrid" automatically pulls in "Button" and "Table".
- **Distribute via Remote URLs:** Deploy the registry to static hosting (GitHub Pages/S3) for organizational use.

## Dependencies

- **React & TypeScript** — Core runtime and type safety for the component source code.
- **Tailwind CSS** — The styling engine used by the shadcn CLI to ensure consistent design tokens.
- **Radix UI** — The accessible primitive layer for complex components (Dropdowns, Dialogs, etc.).
- **Vite** — Recommended development environment for building and previewing the component library.

## When to Use

**Use when:**
- You are building a design system for a large organization with multiple independent React teams.
- You want to provide a "starting point" for components that teams can then customize.
- You need to ensure accessibility (via Radix) but allow local styling overrides via Tailwind.
- You want to avoid the overhead of versioning and publishing hundreds of small NPM packages.

**When NOT to use:**
- You are building a single, small application (just use standard shadcn/ui).
- You require strict, unchangeable UI (an NPM package with compiled CSS is better for "locked" designs).
- Your team is not comfortable owning and maintaining the component source code in their local repo.

## Quick Reference

| Problem | Solution | Keywords |
| ------- | -------- | -------- |
| Component Drift | Use `shadcn add` to sync local code with central registry | `registry`, `sync` |
| Dependency Hell | Define `dependencies` and `registryDependencies` in JSON | `manifest`, `schema` |
| Manual JSON Errors | Automate registry generation with a TS build script | `automation`, `fs` |
| Private Distribution | Host `registry.json` on internal S3 or private GitHub Pages | `hosting`, `distribution` |

## Core Patterns

### Registry Directory Structure

A clean workspace separates the **registry source** (what is distributed) from the **documentation/preview** (Vite app).

**Workspace Layout:**
```text
/
├── apps/
│   └── www/                # Vite/Next.js documentation site
├── registry/               # The source of truth
│   ├── ui/                 # Atomic components (buttons, inputs)
│   ├── internal/           # Complex patterns (data-tables, shells)
│   └── hooks/              # Shared logic
├── scripts/
│   └── build-registry.ts   # The automation engine
├── public/
│   └── r/                  # The generated output (distributed)
│       ├── index.json      # Main entry point
│       └── button.json     # Individual component manifest
└── package.json
```

### Automation: The Registry Builder

**The problem:** Manually writing JSON files for every component is error-prone. If you add a `lucide-react` icon to a component, you might forget to update the `dependencies` array in the JSON.

**❌ Common mistake:** Hardcoding registry entries.
```json
// registry.json (Manual - DON'T DO THIS AT SCALE)
{
  "name": "button",
  "dependencies": ["lucide-react"], 
  "files": ["ui/button.tsx"] // Easy to break if file moves
}
```

**✅ Production pattern:** Use a script to crawl the `registry/` directory and extract metadata.

```typescript
// scripts/build-registry.ts excerpt
import { writeFileSync, readFileSync, readdirSync } from "fs";
import path from "path";

const REGISTRY_PATH = path.join(process.cwd(), "registry");
const OUTPUT_PATH = path.join(process.cwd(), "public/r");

async function buildRegistry() {
  const components = readdirSync(path.join(REGISTRY_PATH, "ui"));
  
  const registryEntries = components.map((file) => {
    const name = file.replace(".tsx", "");
    const content = readFileSync(path.join(REGISTRY_PATH, "ui", file), "utf8");

    return {
      name,
      type: "registry:ui",
      dependencies: extractDependencies(content), // Regex to find imports
      files: [
        {
          path: `ui/${file}`,
          content: content,
          type: "registry:ui"
        }
      ],
    };
  });

  writeFileSync(
    path.join(OUTPUT_PATH, "index.json"),
    JSON.stringify(registryEntries, null, 2)
  );
}
```

**Key insight:** The registry is essentially a "Virtual File System" in JSON format. The CLI reads this JSON and reconstructs the files in the consumer's project.

## Usage Examples

### 1. The Registry Manifest (`index.json`)
The consumer points their CLI to this file. It maps component names to their definitions.

```json
{
  "name": "acme-ui",
  "items": [
    {
      "name": "brand-button",
      "type": "registry:ui",
      "author": "Acme Design Team",
      "dependencies": ["framer-motion", "lucide-react"],
      "registryDependencies": ["button"],
      "files": [
        {
          "path": "ui/brand-button.tsx",
          "content": "import { Button } from '@/components/ui/button'...",
          "type": "registry:ui"
        }
      ]
    }
  ]
}
```

### 2. Consuming the Custom Registry
Once hosted (e.g., at `https://ui.acme.com/r/index.json`), a developer can install a component using:

```bash
# Add a specific component from your custom registry
npx shadcn@latest add https://ui.acme.com/r/brand-button.json
```

## Common Mistakes

| Mistake | Why It's Wrong | Fix |
| ------- | -------------- | --- |
| Absolute Paths | Imports like `import { x } from 'src/components'` will break in the consumer app. | Use relative imports or standard `@/components/ui` aliases. |
| Missing Dependencies | Forgetting to list `lucide-react` in the JSON means the consumer's code won't compile after download. | Use AST parsing or regex in your build script to auto-detect imports. |
| Tailwing Config Drift | Component styles rely on specific Tailwind colors/animations not present in the consumer app. | Include a `tailwind.config.js` snippet in the registry item documentation. |

## Red Flags

- **No Registry Dependencies:** If your "DataCard" uses a "Button" but the registry entry for DataCard doesn't list Button as a `registryDependency`, the user will have broken code.
- **Manual JSON Edits:** If your `public/r/` folder is edited by hand instead of generated, it will eventually become inconsistent.
- **Hardcoded Registry URLs:** Avoid hardcoding the registry URL inside component code; keep components agnostic of where they are hosted.

import fs from 'fs';
import path from 'path';

const registryDir = './registry/ui';
const outputDir = './public/r';

const buildRegistry = () => {
  const files = fs.readdirSync(registryDir);
  const index = files.map(file => {
    const name = path.parse(file).name;
    const content = fs.readFileSync(path.join(registryDir, file), 'utf8');
    
    const item = {
      name,
      type: 'registry:ui',
      dependencies: ['lucide-react'], // Simplified: should be extracted dynamically
      files: [{
        path: `ui/${file}`,
        content,
        type: 'registry:ui'
      }]
    };
    
    // Save individual file for 'shadcn add [url]'
    fs.writeFileSync(path.join(outputDir, `${name}.json`), JSON.stringify(item, null, 2));
    return item;
  });

  // Save index for discovery
  fs.writeFileSync(path.join(outputDir, 'index.json'), JSON.stringify(index, null, 2));
};

buildRegistry();
