# Feature Comparison Matrix (Current vs. Target Architecture)

This matrix highlights exactly where the `qredence/fleet-rlm` repository stands today versus the hyper-advanced capability matrix we are building via our 5-phase Surgical Integration Plan.

| Feature Domain                  | Current `fleet-rlm` (Baseline)                                                                                                                                | Target Post-Integration (Hyper-Advanced)                                                                                                                                                                   |
| :------------------------------ | :------------------------------------------------------------------------------------------------------------------------------------------------------------ | :--------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **Agent Paradigm**              | Single-tier ReAct Agent. Uses a flat `dspy.ReAct` loop that calls a Modal execution tool in a single-pass "Chain of Thought".                                 | **Dual-Loop Multi-Agent**. A `dspy.ReAct` Supervisor handles chat/planning, delegating engineering tasks to a deeply nested, self-correcting `RLMEngine`.                                                  |
| **Code Generation Strategy**    | **One-Shot Generation**. The agent guesses the python code, runs it, and relies on the top-level ReAct loop to realize if it failed.                          | **Recursive Language Model (RLM)**. A dedicated inner loop writes code, executes it, evaluates the Python compiler `stderr`, and rewrites the script iteratively _before_ surfacing to the Supervisor.     |
| **Context Window Security**     | **Vulnerable to Context Rot**. If a Modal sandbox command prints 100,000 rows of an SQL database, the DSPy context window bursts or loses tracking.           | **Absolute Truncation Guards**. `MAX_CHARS=2000` hard-stops at the execution layer. Forces the RLM to adapt to truncation and write actual data-filtering scripts (e.g. `.head()`).                        |
| **Long-Term Memory**            | Memory constrained to the current `dspy.History` message array and transient sandbox buffers. Memory dies when the server restarts or context limits are hit. | **Evolutive pgvector Database**. Asynchronous Neon DB storing `TaxonomyNodes` and `AgentMemory`. DSPy tools can natively perform `l2_distance` embeddings search to pull rules from months ago.            |
| **Client Interface**            | Terminal UI (`tui-cli`, `tui-ink`) locked into synchronous textual outputs.                                                                                   | **Dual-Pane React Dashboard**. Multiplexed WebSockets feed Vite/Zustand transient stores. Shows Chat on the left, and a sliding Workspace/Terminal Drawer visualizing the nested RLM loops asynchronously. |
| **System Visibility / Tracing** | Standard stdout debug logs.                                                                                                                                   | **PostHog DSPy Bindings**. Every ReAct thought, tools call, and sub-RLM generation is deterministically traced and telemetry-synced.                                                                       |
